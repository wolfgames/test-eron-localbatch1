# Manifest Contract

Machine-readable contract for valid asset manifests. Use this when generating or validating manifests (AI, tooling, or hand-authored). Type definitions live in `src/core/systems/assets/types.ts`.

---

## 1. Bundle naming rules

### 1.1 Reserved prefixes and semantics

Target is **inferred from bundle name prefix** unless overridden by `target`:

| Prefix   | Loader      | Use case                          |
|----------|-------------|-----------------------------------|
| `boot-`  | DomLoader   | Pre-engine loading screen         |
| `theme-` | DomLoader   | Branding, DOM loading screen      |
| `audio-` | AudioLoader | Sound effects, music              |
| `data-`  | DomLoader   | JSON config files                 |
| `core-`  | GpuLoader   | In-game UI                        |
| `scene-` | GpuLoader   | Gameplay assets                   |
| `fx-`    | GpuLoader   | Particles, effects                |

Bundles without a recognised prefix must set an explicit `kind` field. The loader is determined by `KIND_TO_LOADER` from `@wolfgames/components/core`.

### 1.2 `kind` field (optional)

Bundles may carry an explicit `kind` that classifies them semantically:

| `kind`  | Equivalent prefix | Loader      |
|---------|-------------------|-------------|
| `boot`  | `boot-`           | DomLoader   |
| `theme` | `theme-`          | DomLoader   |
| `audio` | `audio-`          | AudioLoader |
| `data`  | `data-`           | DomLoader   |
| `core`  | `core-`           | GpuLoader   |
| `scene` | `scene-`          | GpuLoader   |
| `fx`    | `fx-`             | GpuLoader   |

**Rules:**
- If `kind` is set, it must be one of the values above.
- If both `kind` and a name prefix are present, they must be consistent (e.g. `kind: "theme"` with name `theme-branding` is valid; `kind: "audio"` with name `theme-branding` is rejected).
- If neither `kind` nor a recognised prefix is present, validation fails. This ensures every bundle has a clear classification even when generated by an LLM.

### 1.3 Allowed characters and format

- **Recommended (strict mode):** `^[a-z][a-z0-9-]*$` — lowercase letter first, then lowercase letters, digits, or hyphens only. No spaces, underscores, or special characters.
- **Convention:** Prefer one-to-one mapping between bundle name and primary asset when a bundle has a single main file (e.g. `theme-branding` → `atlas-branding-wolf.json`).

### 1.4 Uniqueness

- Bundle **names** must be unique within the manifest.
- Asset **aliases** must be unique across all bundles. Aliases are derived from asset paths (filename without extension). Two bundles containing `ui/button.json` and `game/button.json` both produce alias `button`, causing a collision.

## 2. Asset path rules

- Paths are **relative** to `cdnBase` (and optionally `localBase`). No leading slash.
- **No** `..` segments or absolute URLs (no `http://`, `https://`, or `/` at start).
- **Allowed extensions** (by loader/type, see `inferAssetType` in `types.ts`):

  | Extension              | Inferred type   | Notes                          |
  |------------------------|-----------------|--------------------------------|
  | `.json` (not in `audio/`) | spritesheet  | TexturePacker JSON + image    |
  | `.json` (path contains `audio/`) | audio   | Howler sprite definition      |
  | `.png`, `.jpg`, `.jpeg`, `.webp`, `.gif`, `.svg` | image | Raw image  |
  | `.woff`, `.woff2`, `.ttf`, `.otf` | font   | Web fonts                     |
  | other `.json`          | json            | Generic JSON data             |

- Paths in `bundles[].assets` should point to files that exist under the resolved base (cdn or local). Consistency checks can verify existence on disk.

## 3. Manifest shape (reference)

```typescript
interface Manifest {
  cdnBase: string;      // required, e.g. '/assets' or 'https://cdn.example.com/game/assets'
  localBase?: string;   // optional fallback if CDN fails
  bundles: ManifestBundle[];
}

interface ManifestBundle {
  name: string;         // unique, prefer reserved prefix + kebab-name
  assets: AssetDefinition[];
  kind?: BundleKind;    // optional semantic classification (see §1.2)
}

interface AssetDefinition {
  alias: string;        // unique across all bundles
  src: string;          // path relative to cdnBase/localBase
  data?: Record<string, unknown>;  // optional loader-specific data
  type?: AssetType;     // content type hint (see §3.1)
  tags?: string[];      // freeform tags for filtering/generation metadata
}

type AssetType = 'image' | 'json' | 'spritesheet' | 'audioSprite';
type BundleKind = 'boot' | 'theme' | 'audio' | 'data' | 'core' | 'scene' | 'fx';
```

### 3.1 `AssetDefinition.type` (prompt-friendly)

When generating manifests, the `type` field lets generation tooling specify intent explicitly rather than relying on file-extension inference:

| `type`        | Use case                                    | Example `src`          |
|---------------|---------------------------------------------|------------------------|
| `image`       | Single raster/vector image                  | `bg-city.png`          |
| `json`        | Generic JSON data                           | `levels/chapter-1.json`|
| `spritesheet` | TexturePacker atlas (JSON + image pair)      | `atlas-ui-buttons.json`|
| `audioSprite` | Howler audio sprite (JSON + audio files)     | `sfx-gameplay.json`    |

When `type` is omitted, loaders infer type from file extension and bundle kind (existing behavior).

### 3.2 `AssetDefinition.tags`

Freeform string tags for filtering, grouping, or generation metadata. The runtime does not interpret tags; they are available for tooling, queries, and LLM prompts.

```json
{ "alias": "atlas-tiles", "src": "atlas-tiles.json", "type": "spritesheet", "tags": ["gameplay", "essential"] }
```

## 4. Minimal valid example

```json
{
  "cdnBase": "/assets",
  "localBase": "/assets",
  "bundles": [
    {
      "name": "theme-branding",
      "assets": [
        { "alias": "atlas-branding-wolf", "src": "atlas-branding-wolf.json" }
      ]
    }
  ]
}
```

### 4.1 Example with prompt-friendly metadata

```json
{
  "cdnBase": "/assets",
  "bundles": [
    {
      "name": "boot-loading",
      "assets": [
        { "alias": "spinner", "src": "spinner.png", "type": "image" }
      ]
    },
    {
      "name": "audio-sfx",
      "kind": "audio",
      "assets": [
        {
          "alias": "sfx-gameplay",
          "src": "sfx-gameplay.json",
          "type": "audioSprite",
          "tags": ["gameplay", "essential"]
        }
      ]
    },
    {
      "name": "scene-game",
      "assets": [
        { "alias": "atlas-tiles", "src": "atlas-tiles.json", "type": "spritesheet", "tags": ["gameplay"] },
        { "alias": "atlas-ui", "src": "atlas-ui.json", "type": "spritesheet", "tags": ["ui"] }
      ]
    }
  ]
}
```

## 5. Anti-patterns

- **Leading slash on paths:** `"/atlas.json"` — use `"atlas.json"` or `"ui/atlas.json"`.
- **Absolute URL in path:** `"https://cdn.example.com/atlas.json"` — paths must be relative to `cdnBase`.
- **Duplicate bundle names:** two bundles with the same `name`.
- **Duplicate asset aliases:** two assets across bundles that resolve to the same alias (e.g. `ui/button.json` and `game/button.json` both become alias `button`).
- **Invalid bundle name:** spaces, uppercase, or special characters (in strict mode).
- **Mismatched `kind` and prefix:** `kind: "audio"` on a bundle named `theme-branding`.
- **No prefix and no `kind`:** a bundle named `stuff` with no `kind` — validation cannot determine the loader.
- **Parent path segments:** `"../other/atlas.json"`.
- **Empty `assets` array:** allowed by types but usually indicates a mistake; prefer at least one path per bundle.

## 6. Alignment with asset naming (raw and packed)

Generators should follow one consistent set of rules for both **files on disk** and **manifest entries**.

### 6.1 Raw assets (source files before packing)

- Pattern: `{category}-{name}[_{variant}].{ext}` (e.g. `piece-dot.png`, `ui-button_play.png`).
- Categories: `piece-`, `exit-`, `character-`, `bg-`, `item-`, `prop-`, `ui-`, `vfx-`, `sfx-`, `music-`.
- Rules: lowercase, hyphen between category and name, underscores for multi-word names and variants.
- Full spec: [Asset naming convention](../guides/assets/naming-convention.md) and machine-readable [naming-convention.schema.json](../guides/assets/naming-convention.schema.json).

### 6.2 Packed outputs (atlases, audio sprites)

- **Sprite atlases:** `atlas-{category}-{name}.json` (and matching `.png`), e.g. `atlas-branding-wolf.json`, `atlas-tiles-mygame.json`. These appear in `theme-*`, `core-*`, `scene-*`, or `fx-*` bundles.
- **Audio sprites:** `sfx-{name}.json` or `music-{name}.json` (e.g. `sfx-mygame.json`, `music-warehouse-puzzle.json`). Paths often under an `audio/` subfolder in the manifest for correct type inference; alternatively flat names like `sfx-mygame.json` are valid if loader routing is explicit.
- **VFX / data JSON:** `vfx-{name}.json` or other `{category}-{name}.json` under appropriate bundles (`fx-*`, `data-*`).

### 6.3 Bundle names

- Prefer alignment with category/name: e.g. `theme-branding` for branding assets, `audio-sfx` for an SFX sprite bundle, `scene-gameplay` for gameplay atlases.

### 6.4 Path layout

- **Recommended:** Flat under `cdnBase` for small games (e.g. `atlas-branding-wolf.json`, `sfx-ui.json`). For larger games, use subfolders by type: `ui/buttons.json`, `audio/sfx-ui.json`, `scenes/gameplay/entities.json`. Document your choice in the game’s README or config so generators stay consistent.

## 7. Validation

### 7.1 Runtime (fail-fast at init)

When `createScaffoldCoordinatorFromGc` builds the coordinator, it runs `validateManifest()` from `@wolfgames/components/core` on the adapted GC manifest **before** creating the coordinator. Invalid manifests throw immediately with a clear error summary. The GC coordinator also validates internally as a second layer.

Checks performed at runtime:
- `cdnBase` is a non-empty string
- Bundle names are unique and match `^[a-z][a-z0-9-]*$`
- Bundle names have a recognised prefix or explicit `kind`; if both are present they must agree
- Asset aliases are unique across all bundles (prevents resolver collisions)
- Asset `src` paths are relative (no leading `/`, no `..`, no absolute URLs)

### 7.2 CLI (offline / CI)

- **Manifest:** run `bun run check:manifest` to validate the game manifest. Uses shared `@wolfgames/components` validation when available, plus scaffold-specific checks (extensions, prefix recommendations). Use `--strict` for strict bundle name regex. Validate a JSON file with `--path=manifest.json`. Schema: [manifest.schema.json](../schemas/manifest.schema.json).
- **Asset filenames:** run `bun run check:assets` to validate files in `public/assets/` against the naming convention. See [Asset naming convention](../guides/assets/naming-convention.md#validation).
